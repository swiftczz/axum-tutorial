# Axum 教程编写约束

本文件是编写和修改教程章节时必须遵循的规范。新增章节前请通读本文件。

---

## 1. 教程对齐版本

- examples 代码对齐 **release 0.8.9** 版本（工作区 `examples/` 目录）
- 教程 `Cargo.toml` 统一用 **crates.io 版本**：`axum = "0.8"`、`edition = "2024"`
- examples 源码用 `path = "../../axum"` 本地路径依赖，教程做了转换
- 依赖版本取 crates.io 最新版（axum 0.8.9 / axum-extra 0.12 / tower-http 0.6 等）

## 2. 代码与 examples 严格一致

- `## 完整代码` 代码块必须和 `examples/<name>/src/main.rs` **逐字节一致**（除注释差异）
- `Cargo.toml` 代码块必须和 `examples/<name>/Cargo.toml` 的 crate 名字和 features 一致
- 版本号可以用最新版，但 **crate 名字和 features 不能写错**
- `axum::serve(...).await` 后统一加 `.unwrap()`（比 examples 默默丢弃错误更好）

## 3. 按代码复杂度分三档

### 第一档：短代码（< 40 行）

**结构**：Cargo.toml → 完整代码 → 运行 → 解读 → 手写任务 → 小结 → 源码对照

适用：ch01-05 等简单章节。代码短，一次给完没有理解负担。

### 第二档：中等代码（40-80 行）

**结构**：Cargo.toml → 分 2-3 步增量构建 → 完整代码汇总 → 运行 → 手写任务 → 小结 → 源码对照

每步引入 1 个新概念，每步代码完整可运行。最后给完整代码作为汇总。

### 第三档：长代码（100+ 行）

**结构**：Cargo.toml → 分 4-6 步从零构建 → 每步可运行验证 → 完整代码汇总 → 运行 → 手写任务 → 小结 → 源码对照

每步只引入 1-2 个新概念。读者跟着每步敲一遍，理解每一层加进来解决了什么问题。

## 4. "新面孔"框

每个新 API、新概念、新 crate 函数首次出现时，用引用块解释：

```markdown
> **新面孔：`XXX`**
>
> 一句话解释它是什么。然后解释它在本章做什么、为什么需要它。
> 和前面章节学过的东西做对比（如"和 Json<T> 一样是 extractor，但..."）。
```

- **只在该概念第一次出现时加**，后面章节重复出现不需要重复加
- **关注"相比前面章节引入的新东西"**：读者看到新 API 时不应该困惑"这是什么"
- 新概念包括但不限于：新 crate、新 trait、新 derive 宏、新 axum 提取器、新 Tower layer

## 5. 每步的代码要求

- **完整可运行**：每步的代码块是完整的 `main.rs`（不是片段），读者可以 `cargo run` 跑起来
- **增量递进**：每步只在上一步基础上加 1-2 个新东西，不一次塞太多
- **先讲"为什么"再给代码**：每步开头先说"这步要解决什么问题"，然后给代码
- **每步后可以验证**：给出 `curl` 命令或浏览器操作，让读者确认这步做对了

## 6. 分步构建的设计原则

- **从最小可运行版本开始**：先写一个能跑的最简版本（如 GET 返回空数组）
- **逐步叠加功能**：每步加一个功能层（如先 GET → 再 POST → 再 PATCH/DELETE → 最后中间件）
- **新概念在需要时才引入**：不要提前铺垫不需要的东西
- **每步回答"为什么需要这个"**：不是"这个 API 怎么用"，而是"不用它会怎样"

## 7. 不要做的事

- **不要一次给完整代码然后逐行解释**（这是旧模式，读者在完整代码阶段就迷失了）
- **不要省略 use 语句**（除非该步的代码块已包含完整 use）
- **不要把所有概念塞到一步**（如不要在第一步同时引入 State + Arc + RwLock + Json + Path）
- **不要重复解释前面章节已讲过的概念**（如 ch16 讲过 `State<T>`，ch17 就不需要重新解释）
- **不要写冗长的原理铺垫**（如 Rust 的所有权/生命周期基础不属于 axum 教程范围）
- **不要用统一模板套所有章节**（不同章节问题不同，结构应该灵活调整）

## 8. 章节标准结构（分步构建模式）

```markdown
# NN. 章节名

对应示例：`examples/xxx`

一句话说这章做什么，以及相比前面章节引入了什么新东西。

## Cargo.toml

（完整依赖块，axum = "0.8", edition = "2024"）

---

## 第一步：XXX

先说"这步解决什么问题"。

（完整可运行代码块）

> **新面孔：`XXX`**
>
> 解释新概念。

（验证命令：curl / 浏览器操作）

---

## 第二步：XXX

...

---

## 完整代码

（和 examples 逐字节一致的完整代码块）

## 运行

（完整运行和验证命令）

## 手写任务

1-5 个小改动，帮读者确认理解。

## 小结

- 按步骤总结每步引入的新概念
- 和前面章节做对比（如"和 ch16 的区别是..."）

## 源码对照

- `examples/xxx/Cargo.toml`
- `examples/xxx/src/main.rs`
```

## 9. 模板灵活化

不同类型的章节可以用不同的核心结构：

| 章节类型 | 核心结构 | 例子 |
|---|---|---|
| 短代码(< 40行) | Cargo.toml → 完整代码 → 运行 → 解读 → 小结 | ch01-05 |
| 中代码(40-80行) | 最小版 → 增量步骤(2-3步) → 完整版 → 小结 | ch06-15 |
| 长代码(100+行) | 从零构建(4-6步) → 每步可运行 → 完整版 → 小结 | ch16/17/19/42 |
| 概念章(中间件/测试) | 先讲"为什么需要" → 最小例子 → 完整应用 | ch21-27 |
| 对比章(数据库/TLS) | 选型表 → 1 个详细示例 → 其他只讲差异 | ch32-37/53-55 |

## 10. 交叉引用规则

- 引用其他章节用"第 NN 章"格式（如"见第 16 章"）
- 章节重排时必须同步更新所有交叉引用
- 附录 A0 被多个章节引用，章号变化时需要更新

## 11. 文件命名和序号

- 文件名：`NN-name.md`（两位数字 + examples 目录名）
- 附录：`a0-`、`a1-`、`a2-` 前缀
- 序号 01-55 连续，新增/移除章节时需要批量重命名和更新交叉引用
- 当前序号方案保留 testing 提前到 ch21 和 wasm 后移到 ch48
