# Axum 教程编写约束

本文件是编写和修改教程章节时必须遵循的规范。新增章节前请通读本文件。

## 规则速查

| # | 主题 | 一句话 |
| --- | --- | --- |
| 1 | 版本对齐 | examples 对齐 release 0.8.9，教程用 crates.io 版本 |
| 2 | 代码一致 + 完整代码块规则 | Cargo.toml 说明新增依赖；完整代码块各档位都保留（参考版） |
| 3 | 三档结构 | 短(<40行)/中(40-80)/长(100+)，每档结构不同 |
| 4 | "新面孔"框 | 新 API/概念首次出现才解释，后面不重复 |
| 5 | 每步代码要求 | 完整可运行，增量递进，先讲"为什么"再给代码 |
| 6 | 分步设计原则 | 从最小版开始，逐步叠加，每步回答"不用它会怎样" |
| 7 | 禁止做的事 | 不一次塞概念、不重复解释、不冗长铺垫 |
| 8 | 标准骨架 | 默认模板（按档位裁剪） |
| 9 | 模板灵活化 + 对比章特殊要求 | 对比章（DB/TLS）首段必须有跨章节选型表 |
| 10 | 交叉引用 | 用"第 NN 章"格式 |
| 11 | 文件命名 | `NN-name.md`，序号连续 |
| 12 | **防止冗余硬规则** | 代码与解释 1:1、本章内禁止重复、小结 ≤8 行 |

---

## 1. 教程对齐版本

- examples 代码对齐 **release 0.8.9** 版本（工作区 `examples/` 目录）
- 教程 `Cargo.toml` 统一用 **crates.io 版本**：`axum = "0.8"`、`edition = "2024"`
- examples 源码用 `path = "../../axum"` 本地路径依赖，教程做了转换
- 依赖版本取 crates.io 最新版（axum 0.8.9 / axum-extra 0.12 / tower-http 0.6 等）

## 2. 代码与 examples 严格一致

- `Cargo.toml` 代码块必须和 `examples/<name>/Cargo.toml` 的 crate 名字和 features 一致
- 版本号可以用最新版，但 **crate 名字和 features 不能写错**
- `axum::serve(...).await` 后统一加 `.unwrap()`（比 examples 默默丢弃错误更好）
- **`Cargo.toml` 块下方必须有一句话说明：本章相比前面章节新增/启用了哪些依赖**。例：`axum-extra = { features = ["with-rejection"] }` 这种新面孔要在下面用引用块解释为什么本章需要它。

### "完整代码"块的使用规则

所有档位章节都保留 `## 完整代码` 块（**不删除**），原因：

- 分步代码是**教学版**（带 `# use ...` 隐藏行、带讲解、有时简化）
- 完整代码是**参考版**（可直接 `cargo run` 跑、可整段复制粘贴）
- 两者职责不同：分步用于理解，完整代码用于"我要跑起来"和"我手写错了对照"

具体要求：

- 完整代码块必须和 `examples/<name>/src/main.rs` **逐字节一致**（除注释差异）
- 多文件章（如 ch11 有 4 个 .rs）每个文件单独一个代码块，按 `main.rs` → 子模块顺序排列
- 长代码章末尾的完整代码块**前面加一句**"分步代码用于理解，下面是可复制的完整版，建议跑通后再回看"，明确两种代码的不同用途

## 3. 按代码复杂度分三档

### 第一档：短代码（< 40 行）

**结构**：Cargo.toml → 完整代码 → 运行 → 解读 → 手写任务 → 小结 → 源码对照

适用：ch01-05 等简单章节。代码短，一次给完没有理解负担。

### 第二档：中等代码（40-80 行）

**结构**：Cargo.toml → 分 2-3 步增量构建 → 完整代码（参考版，可直接复制） → 运行 → 手写任务 → 小结 → 源码对照

每步引入 1 个新概念，每步代码完整可运行。最后给完整代码作为参考版（与 examples 一致），方便读者复制后整段跑。

### 第三档：长代码（100+ 行）

**结构**：Cargo.toml → 分 4-6 步从零构建 → 每步可运行验证 → 完整代码（参考版，可直接复制） → 运行 → 解读 → 常见问题 → 手写任务 → 小结 → 源码对照

每步只引入 1-2 个新概念。读者跟着每步敲一遍，理解每一层加进来解决了什么问题。完整代码块前明确标注"分步代码用于理解，下面是可复制的完整版"，避免读者以为两者重复。

## 4. "新面孔"框

每个新 API、新概念、新 crate 函数首次出现时，用引用块解释：

```markdown
> **新面孔：`XXX`**
>
> 一句话解释它是什么。然后解释它在本章做什么、为什么需要它。
> 和前面章节学过的东西做对比（如"和 Json<T> 一样是 extractor，但..."）。
```

- **只在该概念第一次出现时加**，后面章节重复出现不需要重复加
- **关注"相比前面章节引入的新东西"**：读者看到新 API 时不应该困惑"这是什么"
- 新概念包括但不限于：新 crate、新 trait、新 derive 宏、新 axum 提取器、新 Tower layer

## 5. 每步的代码要求

- **完整可运行**：每步的代码块是完整的 `main.rs`（不是片段），读者可以 `cargo run` 跑起来
- **增量递进**：每步只在上一步基础上加 1-2 个新东西，不一次塞太多
- **先讲"为什么"再给代码**：每步开头先说"这步要解决什么问题"，然后给代码
- **每步后可以验证**：给出 `curl` 命令或浏览器操作，让读者确认这步做对了

## 6. 分步构建的设计原则

- **从最小可运行版本开始**：先写一个能跑的最简版本（如 GET 返回空数组）
- **逐步叠加功能**：每步加一个功能层（如先 GET → 再 POST → 再 PATCH/DELETE → 最后中间件）
- **新概念在需要时才引入**：不要提前铺垫不需要的东西
- **每步回答"为什么需要这个"**：不是"这个 API 怎么用"，而是"不用它会怎样"

## 7. 不要做的事

- **不要一次给完整代码然后逐行解释**（这是旧模式，读者在完整代码阶段就迷失了）
- **不要省略 use 语句**（除非该步的代码块已包含完整 use）
- **不要把所有概念塞到一步**（如不要在第一步同时引入 State + Arc + RwLock + Json + Path）
- **不要重复解释前面章节已讲过的概念**（如 ch16 讲过 `State<T>`，ch17 就不需要重新解释）
- **不要写冗长的原理铺垫**（如 Rust 的所有权/生命周期基础不属于 axum 教程范围）
- **不要用统一模板套所有章节**（不同章节问题不同，结构应该灵活调整）

## 8. 章节标准结构（分步构建模式）

下面是**默认骨架**，按章节档位灵活裁剪（见第 2 条和第 9 条）。短代码章没有"分步"和"解读"，长代码章没有末尾的"完整代码"块。

```markdown
# NN. 章节名

对应示例：`examples/xxx`

**前置章节**（如果本章依赖前面章节的知识，明确列出，如"前置：第 46 章 TLS"）

一句话说这章做什么，以及相比前面章节引入了什么新东西。

分 N 步：第一步做 XXX，第二步做 XXX，……

相比前面章节新引入：**crate 名、API 名、概念名**（让读者知道这章的新东西在哪）

## Cargo.toml

（完整依赖块，axum = "0.8", edition = "2024"）

（一句话说明本章相比前面章节新增/启用了哪些依赖）

> 本地 `axum` 依赖如何配置见 [项目 README](../../../README.md#运行前提)。

---

## 第一步：XXX

先说"这步解决什么问题"。

（完整可运行代码块。**代码与解释的字数比应大致 1:1**——50 行代码至少配 50 行解释，不能贴段代码草草两句带过）

> **新面孔：`XXX`**
>
> 解释新概念（首次出现才解释，本章后面或后面章节重复出现不解释）。

（验证命令：curl / 浏览器操作，让读者确认这步做对了）

---

## 第二步：XXX

...

---

## [完整代码]

（所有档位都保留。长代码章前面加一句"分步代码用于理解，下面是可复制的完整版，建议跑通后再回看"）

## 运行

（完整运行和验证命令）

## 解读

（**只在分步没讲透的"为什么/选型/对比"上展开**：
- 为什么用 A 不用 B
- 和前面章节的横向对比（如 ch27 metrics vs ch22 tracing 的取舍）
- 选型表、生产实践注意

**禁止重复**：本章分步已经解释过的 API 用法、参数含义、代码流程）

## 常见问题

（Q&A 形式补分步没覆盖的边角场景，如"X 一定比 Y 好吗？")

## 手写任务

1-5 个小改动，帮读者确认理解。

## 小结

- 按步骤总结每步引入的新概念（每步一行）
- 和前面章节做对比（如"和 ch16 的区别是..."）
- 不超过 8 行

## 源码对照

- `examples/xxx/Cargo.toml`
- `examples/xxx/src/main.rs`
```

## 9. 模板灵活化

不同类型的章节可以用不同的核心结构：

| 章节类型 | 核心结构 | 例子 |
|---|---|---|
| 短代码(< 40行) | Cargo.toml → 完整代码 → 运行 → 解读 → 小结 | ch01-05 |
| 中代码(40-80行) | 最小版 → 增量步骤(2-3步) → 完整版 → 小结 | ch06-15 |
| 长代码(100+行) | 从零构建(4-6步) → 每步可运行 → 完整代码（参考版） → 运行 → 解读 → 小结 | ch16/17/19/42 |
| 概念章(中间件/测试) | 先讲"为什么需要" → 最小例子 → 完整应用 | ch21-27 |
| 对比章(数据库/TLS) | **章首放跨章节选型表** → 本章详细示例 → 解读只讲本章相比其他同类的差异 | ch32-37/53-55 |

### 对比章节的特殊要求（ch32-37 数据库 / ch53-55 TLS）

这类章节是**横向对照**关系——多个 example 解决同一类问题（不同 PG 客户端、不同 TLS backend）。要求：

1. **每章首段必须有跨章节选型表**：列出同组所有方案的对比（性能、易用性、依赖、适用场景）。例 ch34 开头放"PostgreSQL 客户端对比表"涵盖 sqlx/tokio-postgres/diesel/diesel-async。
2. **本章聚焦讲一种**：详细分步构建这一种方案。
3. **解读部分只讲差异**：和同组其他方案对比，不重复讲共同概念（如"什么是连接池"只在 ch33 第一次讲）。

## 10. 交叉引用规则

- 引用其他章节用"第 NN 章"格式（如"见第 16 章"）
- 章节重排时必须同步更新所有交叉引用
- 附录 A0 被多个章节引用，章号变化时需要更新

## 11. 文件命名和序号

- 文件名：`NN-name.md`（两位数字 + examples 目录名）
- 附录：`a0-`、`a1-`、`a2-` 前缀
- 序号 01-55 连续，新增/移除章节时需要批量重命名和更新交叉引用
- 当前序号方案保留 testing 提前到 ch21 和 wasm 后移到 ch48

## 12. 防止冗余的硬性规则

为避免章节臃肿，以下规则**强制执行**：

1. **分步代码与解释字数比 ≈ 1:1**：50 行分步代码至少配 50 行解释。代码块后只写一两句话就进入下一段是不合格的——要么精简代码（拆成更小步），要么补充解释。**注意**：此规则只针对分步教学代码，不针对 `## 完整代码` 块（参考版本来就不需要解释，是给读者复制用的）。
2. **本章内禁止重复**：分步代码里解释过的概念，`## 解读` 不能再讲一遍。`## 解读` 只放"分步没讲的横向对比、选型理由、生产实践"。
3. **末尾小结不超过 8 行**：小结是总结不是重述，每步一行 + 一行对比即可。
4. **`## 常见问题` 限 3-5 条**：每条必须是真实疑问（不是凑数）。
5. **跨章节首次出现才解释**：第 N 章讲过的概念（如 `State<T>`、`FromRequest`），第 N+1 章直接用，不重复"新面孔"框。读者可以从交叉引用回查。
6. **删除"为完整而完整"的段落**：如果某段落删掉后读者依然能理解全章，就删掉。质量优先于数量。
